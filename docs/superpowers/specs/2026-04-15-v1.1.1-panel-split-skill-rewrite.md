# v1.1.1: Panel Split + Skill Rewrite

## Context

v1.1 merged the diff/review view and the new Files view into a single right panel with two tabs (Files / Changes). This created UX problems:
- Tab switching inside the panel feels buried; users don't discover the diff view
- The tab bar consumes vertical space inside a narrow panel
- Narrow-screen responsive view inherited a Chat/Changes dual-tab that non-technical users don't need

Separately, the 3 built-in skills (document-processing, data-analysis, writing-assistant) have prompt quality issues:
- They reply in English regardless of user locale
- The `question` tool is never triggered because the prompt uses conditional language ("if ambiguous") that models self-resolve to skip

## Changes

### 1. Panel Split: Separate Files and Diff into Independent Views

**Goal:** Restore diff/review as a standalone header button, keep Files as the other standalone header button. Both share the same right panel area but are mutually exclusive.

**Approach:** Keep the existing `sidePanelTab` state mechanism ("files" | "changes") from layout.tsx. Move the tab switching UI from inside the panel to two independent header buttons. Remove the in-panel tab bar.

**Files affected:**

| File | Change |
|------|--------|
| `session-side-panel.tsx` | Remove outer `<Tabs>` wrapper and `<Tabs.List>`. Render Files or Changes content via `<Show when={sidePanelTab() === "..."}>` driven by state, not by Tabs component. The entire Changes content block (diff viewer, file tabs, file tree, drag-drop) is preserved as-is inside the Show guard. |
| `session-header.tsx` | Restore the diff toggle button deleted in `9ac23078f` (16 lines). Icon: `review` / `review-active`. Calls `view().sidePanel.toggleTab("changes")`. File button keeps calling `toggleTab("files")`. Each button highlights independently: icon and `aria-expanded` both use `opened() && tab() === "..."`. |
| `use-session-commands.tsx` | No change. `review.toggle` (mod+shift+r) and `fileTree.toggle` (mod+\) already point to `toggleTab("changes")` and `toggleTab("files")` respectively. |
| `layout.tsx` | Change `sidePanelTab` default from `"changes"` to `"files"`. |
| `session.tsx` | No narrow-screen changes in this release. Desktop panel split only. Narrow-screen Chat/Changes tab bar, `mobileTab` state, and `mobileChanges` logic remain as-is. |

**Behavior:**
- Clicking the diff button when panel is closed: opens panel in Changes mode
- Clicking the diff button when panel shows Changes: closes panel
- Clicking the diff button when panel shows Files: switches to Changes mode
- Same logic for file button (symmetric). Implemented by existing `toggleTab()`.
- First file output auto-opens the Files panel (existing auto-open logic, unchanged).

### 2. Locale Injection

**Goal:** Make the LLM aware of the user's language so skills can reply in the correct locale.

**Approach:** Read the app's i18n locale from the `oc_locale` HTTP cookie (already set by the frontend on every request, see `language.tsx:32-34`) and pass it into `environment()`.

The `oc_locale` cookie contains the i18n short code (e.g., `"zh"`, `"en"`, `"zht"`). This is the authoritative source of truth for the user's selected language, including manual switches via settings.

**Implementation path:**
1. `system.ts` `environment()` gains a `locale` parameter (default `"en"`)
2. `prompt.ts` reads locale from the HTTP context (cookie) and passes it to `environment()`
3. Output in `<env>` block: `User locale: zh`

**Why not `Intl.DateTimeFormat().resolvedOptions().locale`:** PawWork has a language switcher in settings (settings-general.tsx). The Node.js process locale doesn't reflect manual i18n changes, so it can diverge from the user's actual UI language.

### 3. Skill Rewrite

**Goal:** Rewrite 3 built-in SKILL.md files to reliably trigger the `question` tool and reply in the user's locale.

**Root cause of current failure:** The prompt uses conditional language ("Use question if ambiguous. Skip when the request is already specific.") which models interpret as permission to skip. Superpowers skills succeed because they use mandatory workflow phases with `<HARD-GATE>` blocks.

**New structure for all 3 skills:**

```
---
name: ...
description: ...
---

# Role (one sentence)

<GATE>
ALWAYS use the `question` tool to clarify before acting.
Do NOT proceed with assumptions. Do NOT skip this step.
</GATE>

## Workflow

1. **Clarify** - Use `question` tool. Ask the predefined questions for this skill.
2. **Execute** - Select tools, perform the task.
3. **Verify** - Check output, report result.

## Step 1: Clarify
(2-3 predefined questions specific to each skill)

## Step 2: Execute
(Tool selection table + execution rules)

## Step 3: Verify
(Checklist)

## Language
Reply in the user's locale (shown in system environment as "User locale").
```

**Key decisions:**
- SKILL.md written in English (models follow English prompts more reliably)
- `<GATE>` XML tag for non-bypassable constraint (proven pattern from superpowers)
- Predefined questions per skill, not model-judged "is this ambiguous?"
- Tool name: `question` (not `AskUserQuestion`)
- Language instruction references the locale from system environment (Section 2)

**Per-skill clarification questions:**

The `question` tool requires structured input: `header` (short label), `options` (2-4 choices with label + description), and `multiSelect` flag. Each predefined question below must be written in the SKILL.md as a complete `question` tool call example so the model knows the expected schema.

document-processing:
1. header: "Task type", options: ["Create new", "Edit existing", "Convert format", "Extract content"]
2. header: "Source", options: ["I'll upload/specify files", "Use files from a previous step"]

data-analysis:
1. header: "Data source", options: ["Spreadsheet (xlsx/csv)", "Database export", "I'll describe the data"]
2. header: "Output", options: ["Summary report", "Chart/visualization", "Updated spreadsheet", "All of the above"]

writing-assistant:
1. header: "Content type", options: ["Email", "Report/memo", "Announcement", "Plan/proposal"]
2. header: "Tone", options: ["Formal", "Conversational", "Concise/direct", "Persuasive"]
